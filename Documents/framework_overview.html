1. Introduction

What is Sanra Solutions Framework?

Sanra Solutions Enterprise Framework is a comprehensive, production-ready platform designed to accelerate the development of enterprise applications. It provides a complete foundation with pre-built modules for common enterprise needs - authentication, authorization, batch processing, document management, event handling, and more.

The framework is domain-agnostic, meaning it can be adapted to build applications for healthcare, government services, education, finance, or any other industry that requires robust, secure, and scalable software systems.

Why Use This Framework?

Pre-Built Enterprise Features

Start with authentication, authorization, audit logging, and batch processing already implemented. Focus on your domain logic instead of infrastructure.

Modern Technology Stack

Built on proven, enterprise-grade technologies with long-term support. Uses latest stable versions with active communities and extensive documentation.

Consistent Patterns

Enforces best practices and consistent coding patterns across the codebase. Makes onboarding new developers faster and code reviews easier.

Production Ready

Includes health checks, metrics, logging, and monitoring integration out of the box. Designed for containerized deployment from day one.

Who Should Use This Framework?

  • Development Teams building new enterprise applications who want to skip boilerplate setup
  • Organizations that need consistent architecture across multiple applications
  • Projects with requirements for security, compliance, audit trails, and batch processing
  • Teams transitioning from legacy systems to modern cloud-native architecture

3. Architecture Overview

High-Level Architecture

The framework follows a layered, microservices-ready architecture that separates concerns and allows independent scaling of components.

Key Architectural Decisions

Layered Service Architecture

Each service follows a consistent layered pattern: Controllers handle HTTP requests, Services contain business logic, and Repositories manage data access. This separation makes code testable, maintainable, and allows teams to work on different layers independently.

Event-Driven Communication

Services communicate asynchronously through Apache Kafka. When something significant happens (entity created, status changed, etc.), an event is published. Other services can subscribe to these events and react accordingly. This decouples services and enables real-time integrations.

Stateless Services

All services are designed to be stateless - they don't store session data locally. Session state is managed externally through Redis and JWT tokens. This allows any instance of a service to handle any request, enabling horizontal scaling.

Database Per Service

The backend service and scheduler service each have their own database schema. This isolation prevents coupling between services and allows independent schema evolution.

6. Integration Hub Framework (Library)

What is the Integration Hub?

The Integration Hub Framework is a custom Java library built by Sanra Solutions that you embed into your application as a Maven/Gradle dependency. It provides an annotation-driven, component-based architecture for building data integration pipelines with automatic component discovery, flow orchestration, and built-in resilience patterns.

Library, Not a Service

The Integration Hub is NOT a separate microservice. It's an embedded library that runs within your application. You define integration components using annotations, and the framework automatically discovers, registers, and orchestrates them at runtime.

Core Architecture

@InputSource Components

Annotated classes that read data from external sources. Extend base classes like AbstractFileReader, AbstractSftpReader, or AbstractRestClient.

@Transformer Components

Annotated classes that transform and validate data. Implement doTransform() and doValidate() methods for processing logic.

@OutputDestination Components

Annotated classes that send data to external systems. Extend AbstractSftpWriter, AbstractRestClient, or AbstractDatabaseWriter.

FlowDefinition & Engine

Define integration flows connecting inputs → transformers → outputs. The IntegrationEngine executes flows synchronously or asynchronously.

Framework Components

Core Interfaces

Interface Purpose

ITransformer<I, O> Transforms input type to output type with optional validation
IOutputDestination<T> Sends data to a destination and returns a result status
IMerger<T> Merges data from multiple input sources into a single collection

Resilience Features

  • Retry with Backoff - Configurable max attempts, initial delay, and exponential multiplier
  • Circuit Breaker - Automatically stops calling failing systems after threshold exceeded
  • Error Handler - Centralized error handling with custom handlers per exception type
  • Async Execution - Configurable thread pools for parallel flow execution

7. Test Governance Framework (Library)

What is Test Governance?

The Test Governance Framework is a build-time Java library that enforces and validates test coverage for critical business rules. It uses annotations in production code (@BusinessRule) and test code (@TestsRule) to ensure every important business rule has corresponding test coverage.

Build-Time Tool, Not a Runtime Service

Test Governance runs during your build/CI pipeline, NOT at runtime. The annotations have zero runtime impact. The validation engine scans your codebase, matches business rules to tests, and generates compliance reports - failing the build if critical rules lack test coverage.

Core Capabilities

@BusinessRule Annotations

Mark critical methods in production code with rule IDs, severity levels, owners, and compliance framework tags.

@TestsRule Annotations

Link test methods to business rules by rule ID, documenting which scenarios are covered by each test.

Automated Validation

Scans codebase during build to match business rules with tests. Fails build if critical rules lack coverage.

Compliance Reports

Generates JSON and HTML reports showing coverage by rule, team, and compliance framework (CMS, HIPAA, etc.).

How It Works

Step 1: Annotate Business Rules

Add @BusinessRule annotations to critical methods in your production code:

  • ruleId - Unique identifier for the business rule (e.g., "BR-PAY-001")
  • severity - CRITICAL, HIGH, MEDIUM, or LOW (determines build failure behavior)
  • owner - Team or person responsible for the rule
  • compliance - Applicable frameworks (CMS_SMC, HIPAA, FISMA, etc.)

Step 2: Link Tests to Rules

Add @TestsRule annotations to your test methods:

  • ruleId - References the business rule being tested
  • scenarios - List of scenarios this test covers
  • category - UNIT, INTEGRATION, COMPLIANCE, etc.

Step 3: Run Validation

Validation runs automatically during your build via:

  • Maven Plugin - mvn test-governance:validate
  • JUnit 5 Extension - Runs during test phase with @ValidateGovernance
  • Spring Boot Starter - Auto-configured validation in test scope

Severity Levels & Build Behavior

Severity Build Behavior Use For
CRITICAL Fails build immediately Financial calculations, security, compliance-critical logic
HIGH Fails on PR merge Core business logic, data integrity rules
MEDIUM Warning only Standard business features
LOW Nightly builds only Edge cases, nice-to-have validations

Common Use Cases

  • Compliance Audits - Generate reports proving all critical business rules have test coverage
  • CI/CD Quality Gates - Fail builds when critical rules lack tests
  • Team Ownership Tracking - Reports show coverage by team/owner
  • Framework Compliance - Track coverage against CMS, HIPAA, FISMA requirements
  • Technical Debt Visibility - Identify untested business rules before they cause issues

8. Security Features

Authentication

The framework uses Keycloak as the identity provider, supporting multiple authentication methods:

  • Username/Password - Traditional login with configurable password policies
  • Single Sign-On (SSO) - Integration with enterprise identity providers via SAML or OIDC
  • Multi-Factor Authentication - TOTP (authenticator apps), SMS, or email verification
  • Social Login - Optional integration with Google, Microsoft, etc.
  • API Keys - Service-to-service authentication for integrations

Authorization

Access control is implemented at multiple levels:

Role-Based Access Control (RBAC)

Users are assigned roles (Admin, Manager, Staff, Viewer, etc.) that determine what actions they can perform. Roles can be hierarchical - an Admin inherits all permissions of Manager, Staff, and Viewer.

Resource-Level Permissions

Beyond roles, the framework supports checking if a user "owns" or has access to a specific resource. For example, a case worker can only view cases assigned to them or their team.

Field-Level Authorization

Sensitive fields (SSN, salary, medical info) can be hidden or masked based on the user's role. The same API returns different field visibility depending on who is calling it.

Data Protection

  • Encryption at Rest - Sensitive database columns can be encrypted using AES-256
  • Encryption in Transit - All communication uses TLS 1.3
  • Audit Trails - Every data change is logged with user attribution
  • Soft Deletes - Records are never physically deleted, maintaining data integrity
  • Session Management - Configurable session timeouts, concurrent session limits

Compliance Readiness

The framework includes features that help meet common compliance requirements:

Standard Relevant Features
HIPAA Audit logging, access controls, encryption, session management
SOC 2 Change tracking, access controls, monitoring, incident logging
GDPR Data export capability, soft delete (right to erasure), consent tracking
Section 508 Frontend components follow WCAG guidelines for accessibility

9. Deployment Options

Development Environment

For local development, the framework provides a Docker Compose configuration that starts all required services:

  • PostgreSQL database (main application data)
  • PostgreSQL database (scheduler data)
  • Apache Kafka with Zookeeper (event streaming)
  • Redis (caching and sessions)
  • Keycloak (identity management)

A single command starts the entire infrastructure, and the application connects automatically.

Production Deployment

The framework is designed for containerized deployment and supports:

Cloud Platforms

  • AWS - ECS/Fargate, RDS, MSK, ElastiCache, S3
  • Azure - AKS, Azure Database, Event Hubs, Redis Cache
  • Google Cloud - GKE, Cloud SQL, Pub/Sub, Memorystore

On-Premises

  • Kubernetes cluster (any distribution)
  • Docker Swarm
  • Traditional VM deployment with manual orchestration

Scaling Considerations

  • Horizontal Scaling - All services are stateless and can run multiple instances behind a load balancer
  • Database Scaling - Supports read replicas for query-heavy workloads
  • Caching - Redis caching reduces database load for frequently accessed data
  • Event Processing - Kafka partitioning allows parallel event consumption
