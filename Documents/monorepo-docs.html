üìã Overview

This documentation outlines the workflow management strategy for our monorepo. The primary goals are:

  • Maintain code organization across frontend, backend, and shared components
  • Enforce proper code reviews by the right team members
  • Prevent accidental modifications to unrelated code sections
  • Automate testing and quality checks
  • Ensure clean git history and traceable changes
Important: This is workflow management, not security. All team members can view all code, but the process ensures proper reviews and prevents mistakes.

How CODEOWNERS Works

  • When a PR is created, GitHub automatically requests reviews from code owners
  • At least one code owner must approve before merging (when enabled in branch protection)
  • Code owners are determined by the files changed in the PR

üîí Branch Protection Rules

For main Branch

Navigate to: Settings ‚Üí Branches ‚Üí Add branch protection rule

Protect matching branches

  • ‚úÖ Require a pull request before merging
    • Require approvals: 2
    • Dismiss stale pull request approvals when new commits are pushed
    • Require review from Code Owners
    • Require approval of the most recent reviewable push

Status Checks

  • ‚úÖ Require status checks to pass before merging
    • Require branches to be up to date before merging
    • Required checks:
      • frontend-tests
      • backend-tests
      • lint-check
      • build-check

Additional Settings

  • ‚úÖ Require conversation resolution before merging
  • ‚úÖ Require linear history (optional)
  • ‚úÖ Do not allow bypassing the above settings
  • ‚úÖ Restrict who can push to matching branches
Note: For develop branch, use similar but slightly relaxed settings (1 approval instead of 2).

üîÑ CI/CD Configuration

GitHub Actions Workflow

Create .github/workflows/ci.yml:

name: CI Pipeline

on:
  pull_request:
    branches: 
      - main
      - develop
  push:
    branches:
      - main
      - develop

jobs:
  # Detect changed files
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      frontend: ${{ steps.filter.outputs.frontend }}
      backend: ${{ steps.filter.outputs.backend }}
    steps:
      - uses: actions/checkout@v3
      - uses: dorny/paths-filter@v2
        id: filter
        with:
          filters: |
            frontend:
              - 'frontend/**'
              - 'shared/**'
            backend:
              - 'backend/**'
              - 'shared/**'

  # Frontend Tests
  frontend-tests:
    needs: detect-changes
    if: needs.detect-changes.outputs.frontend == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
      - name: Install & Test
        run: |
          cd frontend
          npm ci
          npm run lint
          npm test
          npm run build
Benefit: CI automatically runs tests only for changed components, saving time and resources.

üíº Developer Workflows

Frontend Developer Workflow

# 1. Start with latest code
git checkout develop
git pull origin develop

# 2. Create feature branch
git checkout -b frontend/add-user-profile

# 3. Work in frontend folder
cd frontend/
# Make your changes...

# 4. Run tests locally
npm test
npm run lint

# 5. Commit changes
cd ..
git add frontend/
git commit -m "feat(frontend): add user profile component"

# 6. Push branch
git push origin frontend/add-user-profile

# 7. Create Pull Request on GitHub
# - Base: develop
# - Reviewers: Auto-assigned via CODEOWNERS

# 8. After approval and CI passes, merge via GitHub UI

# 9. Clean up
git checkout develop
git pull origin develop
git branch -d frontend/add-user-profile

Backend Developer Workflow

# Similar process but in backend folder
git checkout -b backend/add-authentication-api
cd backend/
# Make changes...
git add backend/
git commit -m "feat(backend): implement JWT authentication"
git push origin backend/add-authentication-api
# Create PR on GitHub

Hotfix Workflow

# For urgent production fixes
git checkout main
git pull origin main
git checkout -b hotfix/fix-payment-error

# Make minimal fix and test
git add .
git commit -m "fix: resolve payment processing error"
git push origin hotfix/fix-payment-error

# Create PR to main (requires immediate review)
# After merge, cherry-pick to develop

‚öñÔ∏è GitHub Users vs GitHub Teams

üë§ GitHub Users (Individual Access)

Adding individual users directly to repository.

Pros

  • Simple for very small projects (2-3 people)
  • Direct control over individual permissions
  • No team setup required

Cons

  • Doesn't scale well
  • Can't use CODEOWNERS effectively
  • No logical grouping by role
  • Hard to manage role changes
  • Repetitive for multiple repos
  • No team-level notifications

üë• GitHub Teams (Group Access)

Creating teams and assigning team permissions.

Pros

  • Scalable - manage groups instead of individuals
  • Easy role changes - just move between teams
  • Works great with CODEOWNERS
  • Team-level notifications (@mentions)
  • Organizational clarity
  • Cross-repo consistency
  • Easier onboarding/offboarding

Cons

  • Requires initial team setup
  • Might be overkill for 2-3 person projects

Real-World Scenarios

Scenario 1: New Frontend Developer Joins

With Individual Users

  1. Add user to Repository 1
  2. Add user to Repository 2
  3. Add user to Repository 3
  4. Update CODEOWNERS in all repos
  5. Update documentation
Time: 30+ minutes, Error-prone

With Teams

  1. Add user to @frontend-team
Time: 2 minutes, Automatic everywhere ‚úÖ

Scenario 2: Developer Switches Roles (Frontend to Backend)

With Individual Users

  1. Change role in each repository
  2. Update all CODEOWNERS files
  3. Update documentation
  4. Notify team members
Time: 1+ hour, Many places to update

With Teams

  1. Remove from @frontend-team
  2. Add to @backend-team
Time: 2 minutes, Permissions update everywhere ‚úÖ
Recommendation: For any project with 4+ developers or plans to grow, use GitHub Teams. The initial setup (15 minutes) provides massive long-term benefits in maintainability and scalability.

PR Naming

  • fix: fix bug description
  • refactor: refactor component name
  • docs: update documentation

Code Review Guidelines

  • Review within 24 hours
  • Be constructive and respectful
  • Test changes locally when needed
  • Check for security issues
  • Verify tests are included
